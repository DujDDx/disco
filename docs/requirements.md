# 多硬盘本地存储调度 CLI 需求文档

## 1. 文档信息

- 项目代号：Disco
- 文档类型：产品需求文档（PRD / Requirements）
- 产品形态：本地运行的纯命令行 CLI / TUI 工具
- UI 方式：基于 React 组件化思想实现终端交互界面
- 目标用户：需要把大量文件分散存放在多块本地硬盘中的个人用户或小型工作室

## 2. 项目目标

构建一个本地多硬盘存储调度软件，将多块独立硬盘组织为“硬盘池”，并在不依赖 RAID、NAS、云端或远程数据库的前提下，实现以下能力：

1. 遍历并索引所有硬盘中的文件，即使硬盘未连接，也能通过本地索引搜索到文件所在硬盘与目录位置。
2. 在写入新文件时，根据用户定义的拆分规则（`Solid` / `SolidLayer`）智能调度文件到最合适的硬盘。
3. 在读取文件时，先通过索引定位，再引导用户连接目标硬盘，连接后直接访问文件。
4. 提供终端内可视化，帮助用户理解磁盘占用、目录结构和 `Solid` 标记。
5. 提供智能空间清理能力，在不占用电脑本地磁盘作为交换空间的前提下，尽量整理硬盘池中的空间布局。
6. 采用现代化、模块化、可维护的工程架构，为后续扩展打下基础。

## 3. 产品边界与原则

### 3.1 范围内

- 本地硬盘发现、注册、扫描、索引
- 离线索引检索
- 文件/目录存入硬盘池
- 磁盘挂载状态检测
- 终端可视化展示
- 智能空间调度与清理

### 3.2 范围外

- 不做分布式存储
- 不做网络共享协议
- 不做 RAID、校验盘、纠删码等底层冗余能力
- 不保证操作系统层面的“物理扇区连续”
- 不以内核驱动或文件系统插件方式工作

### 3.3 基本原则

1. 完全本地化：索引、配置、状态全部保存在本地。
2. 可离线检索：硬盘未连接时也能查到“文件在哪块盘、哪个目录”。
3. 安全优先：写入、迁移、清理必须具备中断恢复与结果校验机制。
4. 可解释调度：为什么文件被放到某块硬盘，需要能被用户理解。
5. 规则优先：`Solid` 与 `SolidLayer` 规则高于空间优化策略。
6. 可维护优先：核心能力应模块化拆分，便于测试与后续扩展。

## 4. 核心概念定义

### 4.1 硬盘池（Disk Pool）

由多块被本软件登记的本地硬盘构成的逻辑集合。每块硬盘仍保留自身独立文件系统，不做底层融合挂载。

### 4.2 硬盘标识（Disk Identity）

每块硬盘需要同时具备：

1. 内部唯一标识：用于程序稳定识别同一块硬盘
2. 用户可识别名称：用于 CLI、搜索结果、可视化界面中展示

其中，“内部唯一标识”与“用户可识别名称”必须绑定保存，形成稳定映射关系，便于用户长期识别同一块硬盘。

建议规则如下：

1. 程序内部优先使用设备序列号 / UUID 作为唯一标识
2. 若无法获取，则退化为卷 UUID
3. 若仍无法获取，则使用卷标 + 容量 + 首次登记指纹作为兜底标识
4. 每块硬盘在首次登记时都应生成或确认一个用户可识别名称，例如“素材盘-01”“归档盘-02”
5. 后续所有用户可见界面默认优先显示该名称，并在需要时附带内部唯一标识的简短摘要

说明：

- 内部唯一标识用于避免误识别
- 用户可识别名称用于降低记忆成本
- 即使挂载点变化，或系统卷标变化，只要内部唯一标识匹配，系统仍应识别为同一块硬盘并沿用原有名称

### 4.3 索引目录（Index）

本地维护的文件元数据数据库，至少包含：

- 文件名
- 文件大小
- 所属硬盘
- 所属硬盘名称
- 文件相对路径
- 最后已知挂载点
- 文件哈希（可选）
- 修改时间
- 索引更新时间

### 4.4 `Solid`

用户对某个目录设置的不可拆分标记。被标记为 `Solid` 的目录必须作为一个原子存储单元，不能被拆分到多个硬盘。

### 4.5 `SolidLayer`

用户在“存储模式”下指定的目录拆分深度规则。

定义如下：

| SolidLayer | 含义 | 最小可拆分单元 |
| --- | --- | --- |
| 0 | 整个输入目录不可拆分 | 根目录 |
| 1 | 可拆分一级子目录 | 根目录下的直接子目录 |
| 2 | 可拆分二级子目录 | 第二层子目录 |
| n | 可拆分到第 n 层子目录 | 第 n 层子目录 |
| inf | 可一直拆分到文件 | 单个文件 |

补充规则：

1. `SolidLayer` 只定义“允许拆到多深”。
2. 被显式设置为 `Solid` 的目录优先级更高，即使 `SolidLayer=inf`，该目录仍不可拆分。
3. 如果某个原子单元体积大于当前任何一块可用硬盘的剩余空间，则该任务应失败，并明确提示原因。

### 4.6 挂载状态

- 已连接：目标硬盘当前已挂载且身份匹配
- 未连接：索引中存在该硬盘，但当前未检测到可用挂载点
- 身份冲突：当前插入了同名或同标签硬盘，但身份不匹配

### 4.7 连续可用空间

本项目中的“最大可用连续空间”定义为：

“单块硬盘上可用于容纳一个原子存储单元的最大逻辑剩余空间”。

说明：由于普通用户态程序通常无法控制文件系统底层扇区布局，因此这里不承诺真正的物理连续扇区，只讨论对文件存储有意义的逻辑容量连续性。

## 5. 用户场景

### 5.1 初次建库

用户注册多块硬盘后，执行全量扫描，建立本地索引数据库。之后即使拔掉硬盘，也可以通过索引搜索文件。

### 5.2 离线检索文件

用户搜索某个文件，系统展示文件所在硬盘和目录。如果硬盘未连接，系统提示连接对应硬盘；连接后重新校验硬盘身份，并给出可访问路径。

### 5.3 向硬盘池存入数据

用户进入存储模式，将文件或文件夹拖入终端窗口。CLI 读取拖入的路径，按 `SolidLayer` 和 `Solid` 规则切分原子单元，再根据磁盘剩余空间与调度策略执行存储。

### 5.4 去重写入

用户开启哈希校验后，系统先计算待写入单元的文件哈希，并在索引中查重。若目录中已经存在相同哈希文件，则跳过重复存储，并告知已有位置。

### 5.5 可视化查看

用户通过终端图形化界面查看：

- 硬盘池占用概览
- 各硬盘目录空间占用矩形图
- 文件树结构
- `Solid` 标记状态

### 5.6 空间整理

当至少连接两块硬盘时，用户发起空间清理。系统给出搬移方案，以尽量腾出更大的单盘可用空间，全程不使用电脑本地磁盘作为中转交换区。

## 6. 功能需求

### 6.1 硬盘注册与识别

系统应支持：

1. 手动将硬盘加入硬盘池。
2. 为每块硬盘记录稳定身份标识、用户可识别名称、卷标、容量、首次登记时间和最近挂载点。
3. 在程序启动、扫描、检索、读取前自动检测已连接硬盘。
4. 在硬盘卷标变化但身份一致时仍识别为同一块硬盘。
5. 在检测到身份冲突时阻止误写入。
6. 支持查看和修改用户可识别名称，但不得破坏其与内部唯一标识的绑定关系。

验收标准：

- 同一块硬盘换挂载点后仍能匹配原索引记录。
- 两块同名硬盘同时存在时不会混淆。
- 搜索结果和可视化界面默认显示用户可识别名称，而不是只显示难以辨认的 UUID。

### 6.2 索引构建与维护

系统应支持：

1. 遍历所有已登记硬盘的全部文件。
2. 建立本地索引数据库。
3. 索引在硬盘未连接时仍可被搜索。
4. 支持全量扫描与增量扫描。
5. 支持是否计算哈希的开关：
   - 关闭：仅索引名称、大小、路径等基础元数据
   - 按需：仅在去重或校验时计算
   - 全量：扫描时直接计算
6. 对已删除、已移动、已重命名文件进行索引更新。
7. 对扫描中断具备恢复能力，避免全盘重来。

索引字段至少包括：

| 字段 | 说明 |
| --- | --- |
| entryId | 索引主键 |
| diskId | 所属硬盘 ID |
| diskName | 所属硬盘名称 |
| relativePath | 相对硬盘根目录路径 |
| fileName | 文件名 |
| size | 文件大小 |
| hash | 文件哈希，可为空 |
| mtime | 最后修改时间 |
| type | 文件 / 目录 |
| solidFlag | 是否被标记为 Solid |
| lastSeenMountPoint | 最近挂载点 |
| indexedAt | 最近索引时间 |
| status | 正常 / 缺失 / 未挂载待确认 |

验收标准：

- 未连接硬盘上的文件能被搜索。
- 重新挂载并增量扫描后，索引状态会自动修正。

### 6.3 文件检索与读取

系统应支持：

1. 用户基于名称直接搜索所有硬盘中的文件。
2. 搜索结果展示：
   - 文件名
   - 所属硬盘名称
   - 所属硬盘唯一标识摘要（可选）
   - 文件大小
   - 相对路径
   - 挂载状态
   - 是否有哈希
3. 当目标文件所在硬盘未连接时，提示用户连接指定硬盘。
4. 在检测到目标硬盘已连接后，重新确认身份并输出可访问路径。
5. 若目标文件在多个硬盘中存在副本，应全部展示并标明优先读取项。

推荐交互：

- `search`：搜索索引
- `get`：按结果项定位目标文件

验收标准：

- 搜索到未挂载硬盘上的文件时，能明确告诉用户“请连接哪块硬盘”。
- 连接成功后，无需重新全量搜索即可访问目标文件。

### 6.4 存储模式

系统应支持：

1. 用户进入存储模式后，将文件或文件夹拖入终端窗口。
2. CLI 正确解析拖入终端的路径字符串，兼容空格、转义和多个路径。
3. 用户可为本次存储指定 `SolidLayer`，默认值为 `0`。
4. 系统将输入内容按 `SolidLayer` 和 `Solid` 规则分解为多个原子存储单元。
5. 在正式写入前生成一份调度计划，并展示每个原子单元将写入哪块硬盘。
6. 写入成功后更新索引。
7. 写入失败时保留可恢复状态，不产生“索引已写入但文件未落盘”的不一致。

默认假设：

- “存入硬盘池”默认表示复制到硬盘池，不自动删除源文件。
- 如需“搬入后删除源文件”，应作为后续增强功能，不纳入当前基础需求。

### 6.4.1 原子存储单元切分规则

1. 输入为单个文件时，原子单元就是该文件本身。
2. 输入为目录时，按 `SolidLayer` 生成最小可拆分单元。
3. 任意被设置为 `Solid` 的目录必须直接升级为原子单元。
4. 原子单元不可跨盘写入。

示例：

- 目录 `A`，`SolidLayer=0`：`A` 整体只能放入一块盘
- 目录 `A`，`SolidLayer=1`：`A/x`、`A/y`、`A/z` 可分别落到不同硬盘
- 目录 `A`，`SolidLayer=inf`：可以按单个文件分配
- 若 `A/x` 被标记为 `Solid`，则 `A/x` 整体不可再拆分

### 6.4.2 写入前查重

当用户开启哈希校验时，系统应支持：

1. 在写入前计算待存文件哈希。
2. 在索引中检索相同哈希。
3. 若已存在相同哈希文件，则跳过重复写入。
4. 向用户展示“重复文件已存在于哪块硬盘、哪个路径”。

说明：

- 目录级别查重以最终叶子文件为单位。
- 当哈希关闭时，不基于文件名或大小自动判重，以避免误判。

### 6.4.3 磁盘选择策略

“最合适的硬盘”至少应满足以下顺序规则：

1. 必须满足原子单元容量需求。
2. 必须满足 `Solid` / `SolidLayer` 约束。
3. 优先减少不可用碎片，尽量提高整体空间利用率。
4. 在多个候选盘都可写时，优先选择写入后剩余空间更接近阈值的硬盘（Best Fit 思路），保留更大的空盘给更大的单元。
5. 对同一批写入任务，尽量减少不必要的跨盘分裂，但不能违反空间最大化目标。

建议实现为可插拔策略，以便后续替换为更复杂的规划算法。

验收标准：

- 当一组目录可以拆分时，系统能自动把它们分配到最合适的硬盘。
- 当一组目录不可拆分且没有单盘能容纳时，系统明确失败，不进行部分写入。

### 6.5 `Solid` 标记管理

系统应支持：

1. 对任意已索引目录设置 `Solid`。
2. 取消某目录的 `Solid` 标记。
3. 在索引和可视化中持久化展示 `Solid` 状态。
4. 在后续存储、迁移、清理中始终遵守 `Solid` 规则。

验收标准：

- 已标记的目录在树形图与磁盘占用图中有明确标识。
- 清理与写入时不会打破 `Solid` 目录原子性。

### 6.6 可视化能力

系统应提供终端内基础可视化能力，包括：

1. 文件树视图
2. 磁盘占用图视图

### 6.6.1 文件树视图

要求：

- 以树形结构展示硬盘池与目录层级
- 展示每个节点大小、挂载状态、`Solid` 标记
- 支持折叠 / 展开
- 支持按硬盘过滤
- 支持在树内快速搜索节点

### 6.6.2 磁盘占用图视图

要求：

- 以矩形平面图展示各硬盘容量占用
- 同一硬盘内继续按目录树层级展示空间占用
- 允许使用方向键 `↑ ↓ ← →` 在可见矩形间移动焦点
- 按空格进入当前目录，查看其子目录的占用分布
- 支持返回上一级
- 当前选中项应显示：
  - 目录名
  - 大小
  - 所属硬盘
  - `Solid` 状态
  - 是否已挂载

验收标准：

- 用户仅通过键盘即可在占用图中完成选择、进入、返回。
- `Solid` 目录在图中有明显视觉标记。

### 6.7 搜索功能

系统应支持：

1. 通过关键字搜索所有硬盘中的任意文件。
2. 至少支持文件名模糊匹配。
3. 可选支持：
   - 精确路径搜索
   - 按大小过滤
   - 按扩展名过滤
   - 按哈希搜索
4. 搜索结果可跳转到：
   - 读取定位
   - 在树中定位
   - 在占用图中定位

性能目标建议：

- 常规查询在本地索引规模 100 万条以内时，响应时间应接近实时。

### 6.8 智能空间清理

系统应支持：

1. 仅在至少连接两块硬盘时启用该功能。
2. 在不使用电脑本地磁盘作为交换空间的前提下，在硬盘之间直接搬移数据。
3. 根据当前空间分布生成清理计划。
4. 清理目标优先为：
   - 腾出更大的单盘可用空间
   - 提高后续大文件/大目录的可放置概率
   - 尽量减少目录碎片化
5. 清理时必须遵守：
   - `Solid` 规则
   - 原子存储单元不可拆分
   - 仅允许“盘 A -> 盘 B”直接复制并删除源数据的安全迁移
6. 正式执行前必须先展示计划预览。
7. 用户确认后才执行实际迁移。
8. 执行中若中断，系统应能识别已完成和未完成的迁移任务。

重要约束：

- 不允许把数据先搬到电脑系统盘、缓存盘或临时目录作为中转。
- 不允许为了整理空间而打破已经存在的 `Solid` 目录边界。

验收标准：

- 至少连接两块硬盘时才能进入清理模式。
- 执行清理后，最大单盘可用空间应不小于清理前。

## 7. CLI 交互需求

本产品需要同时支持“命令模式”和“交互模式”。

### 7.1 命令模式

用于脚本化、单次执行、自动化调用。

建议命令：

```bash
disco disk add <mount-point>
disco disk list
disco scan --all
disco scan --disk <disk-id>
disco search <keyword>
disco get <entry-id>
disco store <path...> --solid-layer=0
disco solid set <indexed-path>
disco solid unset <indexed-path>
disco cleanup --plan
disco cleanup --execute
disco visualize
```

### 7.2 交互模式

用于终端内可视化和复杂操作流程。

要求：

1. 基于 React 组件化组织 UI。
2. 支持列表、树、状态栏、面板布局等终端 UI 组件。
3. 支持键盘导航与操作反馈。
4. 支持长任务进度显示：
   - 扫描进度
   - 哈希计算进度
   - 复制进度
   - 清理进度
5. 支持错误提示、确认对话、计划预览、结果摘要。

## 8. 数据持久化需求

系统至少需要持久化以下数据：

1. 硬盘注册信息
   - 内部唯一标识与用户可识别名称的绑定关系
2. 文件索引数据库
3. `Solid` 标记
4. 操作日志
5. 未完成任务状态
6. 用户配置

建议数据组织：

- 嵌入式数据库：用于文件索引和任务状态
- 配置文件：用于用户偏好和策略设置
- 日志文件：用于调试、审计和故障恢复

持久化要求：

1. 程序异常退出后不应破坏索引一致性。
2. 关键写操作应采用事务或等效机制。
3. 索引更新与文件实际落盘状态必须一致。

## 9. 非功能需求

### 9.1 性能

1. 支持大规模文件索引。
2. 搜索应明显快于重新扫描磁盘。
3. 哈希计算与复制应采用流式处理，避免大文件占满内存。
4. 长任务应支持进度反馈和取消。

### 9.2 可靠性

1. 所有迁移、写入、删除动作都要具备失败检测。
2. 发生断电、拔盘、进程崩溃时，应能恢复到可诊断状态。
3. 程序不得因单块盘离线而导致整体索引不可用。

### 9.3 可维护性

1. 采用模块化架构，避免业务逻辑耦合在 CLI 组件中。
2. 核心调度算法、索引层、UI 层、文件系统适配层应彼此解耦。
3. 关键模块必须具备单元测试。
4. 存储流程和清理流程应有集成测试。
5. 终端交互流程应具备端到端测试能力。

### 9.4 可扩展性

1. 后续应能增加新的调度策略而不重写 CLI。
2. 后续应能增加新的索引字段而不大幅破坏旧数据。
3. 后续应能支持更多可视化视图或导出能力。

### 9.5 可用性

1. 关键错误信息必须可读，不允许只输出系统异常栈。
2. 写入、删除、清理等高风险操作必须二次确认或支持预览。
3. 当用户拖入路径格式不正确时，应给出明确提示。

## 10. 推荐工程架构

为满足“React CLI + 高可维护性”的要求，建议采用分层架构：

### 10.1 模块划分

1. `cli-app`
   - 终端入口
   - 命令解析
   - React TUI 组件

2. `core-domain`
   - 硬盘池模型
   - 索引实体
   - `Solid` / `SolidLayer` 规则
   - 调度计划模型

3. `index-service`
   - 扫描
   - 增量更新
   - 哈希计算
   - 索引查询

4. `planner-service`
   - 写入规划
   - 磁盘选择策略
   - 清理规划

5. `execution-service`
   - 实际复制
   - 校验
   - 中断恢复
   - 事务性状态切换

6. `storage-adapter`
   - 文件系统访问
   - 挂载检测
   - 硬盘身份识别

7. `persistence`
   - 数据库访问
   - 配置读写
   - 日志

### 10.2 架构约束

1. UI 层不得直接操作文件系统。
2. 调度策略不得依赖具体 UI 组件。
3. 文件系统访问必须经过适配层，以便未来适配不同平台。
4. 所有高风险操作必须通过统一任务执行器执行，以保证状态一致。

## 11. 测试要求

至少需要覆盖以下测试层次：

1. 单元测试
   - `SolidLayer` 切分逻辑
   - `Solid` 覆盖规则
   - 磁盘选择策略
   - 去重判断逻辑

2. 集成测试
   - 全量扫描到索引入库
   - 存储写入后索引更新
   - 读取前挂载检查
   - 清理迁移流程

3. 端到端测试
   - 搜索未挂载文件并提示插盘
   - 拖入目录并按 `SolidLayer` 存储
   - 在可视化界面中用方向键导航

## 12. 风险与难点

1. 硬盘身份识别不稳定会导致误判，需要优先解决。
2. 大规模扫描和哈希计算会带来性能压力，需要可恢复、可中断。
3. 清理算法容易复杂化，建议先实现“计划预览 + 人工确认”的安全模式。
4. 终端可视化中的矩形占用图需要在字符界面中做信息密度和可读性的平衡。
5. “拖入终端”本质上是路径输入，需要兼容不同 shell 的转义格式。

## 13. 版本建议

### 13.1 MVP（第一阶段）

建议先交付以下能力：

1. 硬盘注册
2. 全量扫描与离线索引
3. 搜索与读取定位
4. 存储模式
5. `Solid` / `SolidLayer`
6. 哈希去重（可选）
7. 基础树形视图

### 13.2 第二阶段

1. 磁盘占用矩形图
2. 增量扫描
3. 更丰富的搜索过滤条件
4. 写入策略优化

### 13.3 第三阶段

1. 智能空间清理
2. 更复杂的整理策略
3. 更完善的恢复与审计能力

## 14. 当前默认假设

因当前需求尚未进一步细化，本文默认以下前提：

1. 产品优先面向单机本地使用，不考虑多用户并发。
2. 存储动作默认是“复制到硬盘池”，不是“移动并删除原文件”。
3. 搜索主键以文件名为主，哈希搜索为增强能力。
4. 清理目标中的“连续空间”按单盘逻辑可用空间理解，不做物理扇区连续保证。
5. 终端 UI 需要同时支持命令式调用和交互式界面，而不只是纯文本输出。

## 15. 一句话总结

Disco 的核心不是“把多块盘拼成一块盘”，而是“在保持独立硬盘可管理性的前提下，用离线索引、可解释调度和终端可视化，把多硬盘文件管理变成一个可搜索、可规划、可维护的系统”。
